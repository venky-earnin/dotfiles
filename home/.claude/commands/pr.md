description: Open a pull request from the current branch using global PR-style rules
argument-hint: [PR title]
---

Open a PR from the current branch following the **Pull request descriptions**
section in `~/.config/agents/AGENTS.md`. Title is a Conventional Commit
subject; body uses Summary / Validation / optional Notes or Risk.

Reviewer audience = distributed teammates who only see `main` and the feature
branch. The description must stand on its own without local-session context.

Workflow:

1. Verify branch is pushed (push with `-u` if not).
2. Inspect commits since base: `git log <base>..HEAD --oneline`.
3. Inspect diff stat: `git diff --stat <base>..HEAD`.
4. Draft a Conventional Commit title (`type(scope): subject`) — use
   `$ARGUMENTS` as a hint if provided, otherwise derive from the commits.
5. Draft **Summary** in 1-3 bullets focused on *what* changed and *why*.
   Reviewer-facing wording, not a commit recap.
6. Draft **Validation**:
   - Mention added/updated tests in one line — do **not** paste local
     pytest output (CI runs them).
   - Include live job IDs / ARNs / run URLs when applicable: SageMaker,
     HyperPod, Databricks, MLflow, W&B, Airflow, GitHub Actions.
   - Include `before → after` metrics vs. a named baseline when the change
     is supposed to move a metric (paste the numbers, not screenshots).
   - Name the environment for any manual verification (prod / staging /
     ephemeral / local).
   - For schema/config/infra changes, include dry-run output (`bundle
     validate`, `terraform plan`, migration preview).
7. Add **Risk** or **Notes** only if reviewer-relevant.
8. Strip before posting: `/tmp` paths, worktree paths, session IDs,
   "generated by [tool]" language, failed-attempt narratives, debugging
   diaries, internal Slack/DM snippets.
9. Open the PR with `gh pr create --title "..." --body "$(cat <<EOF ... EOF)"`.
10. Report the PR URL to the user.
